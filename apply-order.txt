    (without this, every Kiali graph/health request 403s against the Thanos
     querier in openshift-monitoring — Kiali's external_services.prometheus
     uses use_kiali_token: true, which needs cluster-monitoring-view)

    -- Enable user-workload monitoring (cluster-admin, one-time) --
    -- Required here, not just in Chapter 7: Kiali's traffic graphs (Lab 5b
    -- onward) read istio_requests_total from Thanos, which only has data if
    -- something is scraping the Envoy sidecars into user-workload Prometheus --
    oc patch configmap cluster-monitoring-config -n openshift-monitoring \
      --type=merge -p '{"data":{"config.yaml":"enableUserWorkload: true\n"}}'

    >> Wait: oc get pods -n openshift-user-workload-monitoring -w
    >>       (prometheus-user-workload-0/1 and thanos-ruler-user-workload-0 Running)

    -- Grant the monitoring SA permission to scrape FinanceFlow --
    oc adm policy add-role-to-user \
      view \
      system:serviceaccount:openshift-user-workload-monitoring:prometheus-user-workload \
      -n financeflow-workshop

47. chapters/05-service-mesh/manifests/podmonitor-istio-sidecar.yaml
    (scrapes the Envoy sidecar's /stats/prometheus on port http-envoy-prom —
     without this, Kiali graphs stay empty no matter how much traffic you
     generate, since no istio_requests_total metric ever reaches Thanos)
